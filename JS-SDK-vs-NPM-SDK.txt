NavTalk JS SDK 与 NPM SDK 梳理
生成日期：2026-06-11

一、背景

你现在已经完成的 navtalk-web-sdk 是一个通过 npm 发布的 Web SDK。
从当前项目结构看，它使用 TypeScript + Vite 开发，核心入口是 src/index.ts，打包后通过 dist/navtalk-web-sdk.es.js 和 dist/navtalk-web-sdk.umd.js 对外分发。

当前 SDK 的核心能力包括：

1. 注册 Web Component：my-navtalk-component
2. 通过 license + character-name / character-id 初始化角色
3. 拉取角色头像和 provider 信息
4. 建立 WebSocket 连接
5. 建立 WebRTC 音视频通话
6. 采集本地音频并上传
7. 接收远端视频流并展示
8. 支持 Function Call，并通过 sdkFuncationCall 事件抛给外部页面
9. 支持 UI 样式、按钮图标、按钮文本、默认头像等自定义配置

接下来要做的“相同功能的 JS SDK”，建议理解为：

在不要求用户使用 npm install、不要求用户配置构建工具的前提下，让用户通过 script 标签直接引入一个 .js 文件，然后在普通 HTML 页面中使用 NavTalk 能力。


二、什么是 JS SDK

JS SDK 通常指 JavaScript SDK，是一组用 JavaScript 封装好的能力接口或组件，供网页开发者直接接入某个产品、平台或服务。

在你的场景里，NavTalk JS SDK 可以理解为：

一个可以直接在浏览器页面中运行的 JavaScript 文件，开发者只需要在 HTML 中引入它，就可以使用 NavTalk 的数字人通话、语音交互、WebSocket、WebRTC、Function Call 和 UI 组件能力。

典型使用方式可能是：

方式一：script 直引 + Web Component

<script src="https://cdn.xxx.com/navtalk-web-sdk.umd.js"></script>

<my-navtalk-component
  license="******"
  character-id="******"
  style="width:320px;height:600px;left:100px;top:100px;position:fixed;"
></my-navtalk-component>

方式二：script 直引 + 全局对象

<script src="https://cdn.xxx.com/navtalk-js-sdk.js"></script>
<script>
  NavTalk.init({
    license: "******",
    characterId: "******"
  });
</script>

如果继续沿用你现在的 Web Component 方案，JS SDK 的重点不是重写业务逻辑，而是把现有 npm SDK 打包成更适合浏览器直接使用的产物，并补齐 CDN 引入、全局变量、版本管理、示例 HTML 和兼容性说明。


三、NPM SDK 是什么

NPM SDK 是通过 npm 包管理器分发的 SDK。

开发者使用时通常需要：

1. 项目中安装 Node.js 和 npm / yarn / pnpm
2. 执行 npm install navtalk-web-sdk
3. 在工程代码中 import SDK
4. 通过 Vite、Webpack、Rollup、Next.js、Vue CLI 等构建工具完成打包

当前 npm SDK 的典型使用方式是：

import "navtalk-web-sdk";

然后在页面或组件模板中使用：

<my-navtalk-component
  license="******"
  character-id="******"
></my-navtalk-component>

它更适合 React、Vue、Angular、Next.js、Vite、Webpack 等现代前端工程项目。


四、JS SDK 和 NPM SDK 的相同点

1. 核心功能相同

两者都应该提供同一套 NavTalk 能力：

- 数字人展示
- 点击按钮开始/结束通话
- WebSocket 连接
- WebRTC 音视频连接
- 本地音频采集
- 远端视频播放
- 角色信息获取
- Function Call 配置与事件回调
- 自定义 UI 样式、按钮图标、按钮文本、默认头像

2. 业务流程相同

两者的核心流程都应该是：

- 页面加载 SDK
- 创建或渲染 NavTalk 组件
- 读取 license 和角色配置
- 请求角色详情接口
- 用户点击 Call
- 建立 WebSocket
- 建立 WebRTC
- 开始音频采集
- 接收远端音视频
- 根据连接状态更新 UI
- 收到 Function Call 后通知业务页面

3. 底层浏览器能力相同

两者都依赖浏览器原生能力：

- Web Components / Custom Elements
- Shadow DOM
- fetch
- WebSocket
- WebRTC / RTCPeerConnection
- MediaDevices / getUserMedia
- HTMLVideoElement
- CustomEvent

所以无论是 npm SDK 还是 JS SDK，都主要运行在浏览器环境中，不适合作为纯 Node.js 后端 SDK 使用。

4. 配置参数可以保持一致

建议 JS SDK 与 NPM SDK 尽量保持相同参数命名和含义，例如：

- license
- character-name
- character-id
- websocketUrl
- getInfoUrlByCharacterName
- getInfoUrlByCharacterId
- isRememberPreviousConversation
- function-calls
- default-avatar-background
- playButton_style
- playButton_icon_style
- playButton_text_style
- custom-play-off-icon
- custom-play-connecting-icon
- custom-play-on-icon
- custom-play-off-text
- custom-play-connecting-text
- custom-play-on-text

这样可以降低维护成本，也方便用户从 JS SDK 升级到 NPM SDK，或从 NPM SDK 切换到 JS SDK。

5. 可以复用同一套源码

从当前项目看，JS SDK 没必要重新写一套业务代码。

推荐复用现有 TypeScript 源码：

- src/index.ts
- src/SingleClass/NavTalkManager.ts
- src/SingleClass/WebSocketManager.ts
- src/SingleClass/WebRTCManager.ts
- src/SingleClass/AudioManager.ts
- src/SingleClass/NotificationManager.ts
- src/Style/style.css
- src/assets/*

然后通过不同构建目标输出不同分发文件：

- npm 使用 ES Module / UMD 文件
- JS SDK 使用浏览器 script 可直接加载的 UMD / IIFE 文件


五、JS SDK 和 NPM SDK 的不同点

1. 分发方式不同

NPM SDK：

- 发布到 npm registry
- 用户通过 npm install navtalk-web-sdk 安装
- 版本由 package.json 管理
- 依赖用户项目里的构建工具

JS SDK：

- 发布为一个或多个 .js 文件
- 可以放到 CDN、官网、对象存储、私有服务器
- 用户通过 script 标签引入
- 不要求用户安装 npm，也不要求有构建工具

2. 使用门槛不同

NPM SDK 更适合工程化项目。

用户通常需要理解：

- npm install
- import
- 前端构建
- TypeScript 类型
- 模块化
- 打包产物

JS SDK 更适合快速接入和非工程化页面。

用户只需要理解：

- 在 HTML 中加 script
- 在 HTML 中写 my-navtalk-component
- 填入 license 和 character-id

3. 加载方式不同

NPM SDK：

import "navtalk-web-sdk";

JS SDK：

<script src="https://cdn.xxx.com/navtalk-js-sdk.min.js"></script>

NPM SDK 是模块化加载，JS SDK 是浏览器全局加载。

4. 暴露方式不同

当前 NPM SDK 主要通过副作用注册 Web Component：

import "navtalk-web-sdk";

引入后自动注册：

customElements.define("my-navtalk-component", MyNavTalk)

JS SDK 除了继续自动注册 Web Component，还可以额外暴露全局对象，例如：

window.NavTalk

这样可以支持更传统的写法：

NavTalk.init(...)
NavTalk.destroy(...)
NavTalk.version

是否需要全局对象，取决于你希望 JS SDK 是“组件型 SDK”还是“API 型 SDK”。

建议第一版 JS SDK 保持简单：

- 保留 my-navtalk-component
- 自动注册组件
- 可选暴露 window.NavTalk.version
- 暂时不强行重构成复杂 API

5. 类型支持不同

NPM SDK：

- 可以提供 dist/index.d.ts
- TypeScript 项目可以获得类型提示
- IDE 可以提示参数、方法、事件类型

JS SDK：

- 普通 script 标签没有天然 TypeScript 类型提示
- 主要依赖文档说明
- 如果需要，也可以额外提供 navtalk-js-sdk.d.ts，但不是用户使用的必需条件

6. 依赖处理不同

NPM SDK：

- 可以把依赖声明在 dependencies / peerDependencies
- 用户项目打包时统一处理依赖

JS SDK：

- 最好打成一个自包含文件
- 用户引入一个 JS 文件即可使用
- 外部依赖越少越好
- 图片、CSS 等资源需要考虑内联、base64、相对路径或 CDN 路径

你的当前项目中 CSS 通过 ?raw 内联进 JS，图片由 Vite 处理成构建资源。
如果做 JS SDK，需要重点确认：

- script 引入后图片路径是否正确
- CSS 是否已经内联
- dist 里的 assets 是否需要一起上传
- CDN 路径是否能正确加载资源

7. 版本更新方式不同

NPM SDK：

- 用户通过 npm update 或修改 package.json 升级
- 可以锁定具体版本
- 适合工程项目稳定依赖

JS SDK：

- 如果用户引用固定版本 CDN，升级需要改 URL
- 如果用户引用 latest，可能自动升级但风险更高
- 建议使用固定版本路径，例如：

https://cdn.navtalk.ai/sdk/js/0.0.13/navtalk-js-sdk.min.js

不建议生产环境直接引用：

https://cdn.navtalk.ai/sdk/js/latest/navtalk-js-sdk.min.js

8. 缓存策略不同

NPM SDK：

- 由 npm 包版本和用户构建产物控制缓存

JS SDK：

- 浏览器和 CDN 会缓存 JS 文件
- 文件名最好带版本号或 hash
- 新版本发布后要避免用户加载到旧文件

建议：

- navtalk-js-sdk@0.0.13.js
- navtalk-js-sdk@0.0.13.min.js
- 或 /0.0.13/navtalk-js-sdk.min.js

9. 调试方式不同

NPM SDK：

- 用户可以在工程源码、构建日志、dev server 中调试
- sourcemap 更容易接入

JS SDK：

- 用户直接在浏览器 Console 调试
- 需要提供非压缩版本方便排查
- 建议同时提供：

navtalk-js-sdk.js
navtalk-js-sdk.min.js
navtalk-js-sdk.js.map

10. 适用用户不同

NPM SDK 适合：

- React / Vue / Angular / Next.js 项目
- 有前端工程化能力的团队
- 需要 TypeScript 类型提示的团队
- 希望把 SDK 纳入项目依赖管理的团队

JS SDK 适合：

- 普通 HTML 页面
- 官网、落地页、活动页
- WordPress、Shopify、Webflow 等低代码/建站平台
- 后端模板页面，例如 PHP、Java、Rails、Django 模板
- 只想复制一段 script 快速接入的用户


六、建议的 JS SDK 产品形态

建议第一版 JS SDK 做成“浏览器直引版 Web Component SDK”。

也就是说：

用户这样使用：

<script src="https://cdn.navtalk.ai/sdk/js/0.0.13/navtalk-js-sdk.min.js"></script>

<my-navtalk-component
  license="******"
  character-id="******"
  style="width:320px;height:600px;position:fixed;right:24px;bottom:24px;"
></my-navtalk-component>

这个方案的优点：

- 最大化复用当前 npm SDK
- 开发成本低
- API 学习成本低
- 文档可以和 npm SDK 保持一致
- 用户不需要 npm install
- 适合快速集成和商业推广

后续第二阶段可以再增加 API 型写法：

NavTalk.create({
  container: "#navtalk",
  license: "******",
  characterId: "******"
});


七、对当前项目的改造建议

1. 保留当前 npm SDK

当前 package.json 已经配置：

- main: dist/navtalk-web-sdk.umd.js
- module: dist/navtalk-web-sdk.es.js
- types: dist/index.d.ts

这适合继续作为 npm 包发布。

2. 增加 JS SDK 构建产物

可以在 Vite 中增加或调整构建输出，让 UMD/IIFE 文件更适合 CDN 直接使用。

建议输出：

- dist/navtalk-js-sdk.js
- dist/navtalk-js-sdk.min.js
- dist/navtalk-js-sdk.js.map

3. 确认全局变量名

当前 vite.config.ts 中 name 是：

MyNavTalkSDK

如果作为 JS SDK，建议统一为更清晰的：

NavTalk

或者：

NavTalkSDK

4. 确认 Web Component 标签名

当前标签名是：

my-navtalk-component

正式对外建议考虑改成更品牌化的：

navtalk-widget

或者继续保留 my-navtalk-component 以避免破坏已有用户。

如果已经有用户使用，建议不要直接删除旧标签，可以兼容两个标签。

5. 准备 JS SDK 专用 README

建议单独提供：

- README.md：npm 用户文档
- README-js-sdk.md：script 直引用户文档
- examples/js-sdk/index.html：纯 HTML 示例

6. 注意浏览器兼容性

JS SDK 文档中要说明浏览器要求：

- 支持 Web Components
- 支持 Shadow DOM
- 支持 WebRTC
- 支持 WebSocket
- 支持 microphone permission
- 页面最好运行在 HTTPS 下

尤其是麦克风和 WebRTC，一般要求 HTTPS 或 localhost。


八、总结

一句话概括：

NPM SDK 是给工程化前端项目用的，通过 npm install + import 接入。
JS SDK 是给普通网页直接用的，通过 script 标签接入。

两者的核心业务能力可以完全相同，底层代码也可以大量复用。
真正不同的是分发方式、加载方式、资源路径、类型支持、缓存策略、用户使用门槛和文档示例。

对你的 NavTalk 项目来说，最推荐的方向是：

以当前 npm SDK 为核心源码，增加一个浏览器直引版 JS SDK 产物。
第一版继续沿用 Web Component，用 script 引入后自动注册 my-navtalk-component。
后续再根据用户需求扩展 window.NavTalk.create / init / destroy 等 API。

这样可以最快完成 JS SDK，同时保证 npm SDK 和 JS SDK 的功能一致、维护成本最低。
